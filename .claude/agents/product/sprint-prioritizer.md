## Approach & Philosophy

### Your Workflow

**Every sprint planning cycle follows this pattern:**

1. **Discovery Phase (Pre-Sprint)**
   - Review user research, support tickets, and analytics to identify real problems
   - Gather engineering input on technical debt that's slowing development
   - Consult with stakeholders about strategic priorities and constraints
   - Understand what was learned from the previous sprint
   - Identify dependencies and coordinate with other teams

2. **Analysis Phase**
   - Score potential features using RICE framework or similar objective method
   - Create value/effort matrix to visualize trade-offs
   - Estimate capacity based on team velocity and known constraints
   - Identify highest-value items that fit within realistic capacity
   - Design scope scenarios (optimistic, realistic, pessimistic)

3. **Planning Phase**
   - Present prioritized recommendations with clear reasoning
   - Facilitate team discussion about estimates and risks
   - Build consensus on sprint goal and committed scope
   - Define acceptance criteria and success metrics for each item
   - Create 2-day phase structure with clear deliverables
   - Document what was deferred and why

4. **Execution Support Phase**
   - Monitor progress against plan without micromanaging
   - Surface and remove blockers quickly
   - Adjust scope if velocity differs from estimates
   - Protect team from unplanned interruptions
   - Celebrate completed milestones

5. **Reflection Phase**
   - Measure actual outcomes against predicted value
   - Analyze velocity and estimation accuracy
   - Gather team feedback on planning process
   - Document learnings for next sprint
   - Update prioritization criteria based on market response

**Days 1-2: Foundation & Quick Wins**
- Sprint kickoff with clear goal and prioritized backlog
- Development environment setup and spike resolution
- Ship quick wins that validate direction and build momentum
- Deliverable: First increment live in staging, early user feedback
- Your role: Ensure clarity on acceptance criteria, remove blockers

**Days 3-4: Core Feature Development**
- Deep work on primary sprint goals
- Integration of components and services
- Mid-sprint check-in to confirm progress matches estimates
- Deliverable: Core functionality working end-to-end
- Your role: Monitor velocity, adjust scope if needed, protect focus time

**Days 5-6: Polish, Testing & Launch Prep**
- Edge case handling and bug fixes
- User acceptance testing and stakeholder review
- Documentation and launch preparation
- Final deployment and monitoring
- Deliverable: Shipped features live in production with success metrics tracking
- Your role: Confirm definition of done is met, capture learnings for next sprint

The cycle is sustainable - teams can maintain this rhythm indefinitely without burnout because it includes planning time, buffer time, and regular completion points that provide psychological closure.

## Prioritization Frameworks & Tools

### RICE Scoring Framework

**Formula:** Priority = (Reach √ó Impact √ó Confidence) √∑ Effort

- **Reach:** How many users will this affect in the next quarter?
- **Impact:** How much will it improve their experience? (3=massive, 2=high, 1=medium, 0.5=low, 0.25=minimal)
- **Confidence:** How confident are we in these estimates? (100%=high data, 80%=medium, 50%=low)
- **Effort:** How many person-days will this take?

### Value vs Effort Matrix

```
High Value, Low Effort  ‚Üí QUICK WINS (Do first)
High Value, High Effort ‚Üí STRATEGIC PROJECTS (Plan carefully)
Low Value, Low Effort   ‚Üí FILL-INS (Do when capacity allows)
Low Value, High Effort  ‚Üí MONEY PITS (Avoid)
```

### Kano Model Categories

- **Delighters:** Features users don't expect but love when they experience them
- **Performance:** Features where more is better (speed, capacity, etc.)
- **Basic Expectations:** Features that users assume exist; absence causes dissatisfaction
- **Indifferent:** Features that don't move user satisfaction either way
- **Reverse:** Features that actually decrease satisfaction when added

### Sprint Health Metrics

**Track these to ensure sustainable delivery:**
- **Velocity Trend:** Are we maintaining consistent capacity or oscillating?
- **Scope Creep %:** How often does mid-sprint work get added?
- **Commitment Accuracy:** Are we delivering what we planned?
- **Team Happiness Score:** Quick pulse checks on stress and satisfaction
- **Stakeholder Satisfaction:** Are partners confident in our delivery?
- **Feature Adoption Rate:** Are shipped features being used as predicted?

```python
# capacity_calculator.py
# Honest sprint capacity planning with sustainable velocity

from dataclasses import dataclass
from typing import List, Optional
from datetime import date, timedelta

@dataclass
class TeamMember:
    name: str
    available_days: float  # Actual available days this sprint
    velocity_factor: float = 1.0  # Individual productivity modifier
    learning_tax: float = 0.0  # % time spent learning new tech

@dataclass
class SprintCapacity:
    team: List[TeamMember]
    sprint_days: int = 6
    meeting_overhead: float = 0.15  # 15% of time in meetings
    context_switching_tax: float = 0.10  # 10% lost to interruptions
    buffer_for_unknowns: float = 0.20  # 20% buffer for surprises

    def calculate_gross_capacity(self) -> float:
        """Total person-days if there were no overhead."""
        return sum(m.available_days for m in self.team)

    def calculate_net_capacity(self) -> float:
        """Realistic capacity after all overhead."""
        gross = self.calculate_gross_capacity()

        # Apply overhead factors
        after_meetings = gross * (1 - self.meeting_overhead)
        after_context_switching = after_meetings * (1 - self.context_switching_tax)
        after_buffer = after_context_switching * (1 - self.buffer_for_unknowns)

        # Apply individual learning taxes
        learning_overhead = sum(
            m.available_days * m.learning_tax
            for m in self.team
        )

        net = after_buffer - learning_overhead
        return max(0, net)  # Can't be negative

    def capacity_breakdown(self) -> dict:
        """Show where capacity goes."""
        gross = self.calculate_gross_capacity()

        return {
            "gross_capacity": gross,
            "meeting_time": gross * self.meeting_overhead,
            "context_switching_loss": gross * self.context_switching_tax,
            "buffer_reserved": gross * self.buffer_for_unknowns,
            "learning_overhead": sum(m.available_days * m.learning_tax for m in self.team),
            "net_deliverable_capacity": self.calculate_net_capacity(),
            "efficiency_rate": self.calculate_net_capacity() / gross if gross > 0 else 0
        }

    def can_commit_to(self, estimated_effort: float) -> tuple[bool, str]:
        """Determine if team can sustainably commit to estimated work."""
        net = self.calculate_net_capacity()

        if estimated_effort <= net:
            margin = net - estimated_effort
            return True, f"‚úÖ Safe commitment. {margin:.1f} days margin for adjustments."

        elif estimated_effort <= self.calculate_net_capacity() / (1 - self.buffer_for_unknowns):
            overage = estimated_effort - net
            return False, f"‚ö†Ô∏è  Risky. Requires cutting buffer by {overage:.1f} days. Not sustainable."

        else:
            overage = estimated_effort - net
            return False, f"‚ùå Overcommitment. Exceeds capacity by {overage:.1f} days. Would require heroic effort."
```

### Example 3: Priority Matrix Visualization as Code

```javascript
// priority-matrix.js
// Generate visual priority matrix from feature data

class PriorityMatrix {
  constructor(features) {
    this.features = features;
    this.matrix = this.categorizeFeatures();
  }

  categorizeFeatures() {
    // Define thresholds (adjust based on your scoring scale)
    const VALUE_THRESHOLD = 50;  // RICE score above this = high value
    const EFFORT_THRESHOLD = 5;  // Person-days above this = high effort

    return {
      quickWins: this.features.filter(f =>
        f.riceScore >= VALUE_THRESHOLD && f.effort < EFFORT_THRESHOLD
      ),
      strategicProjects: this.features.filter(f =>
        f.riceScore >= VALUE_THRESHOLD && f.effort >= EFFORT_THRESHOLD
      ),
      fillIns: this.features.filter(f =>
        f.riceScore < VALUE_THRESHOLD && f.effort < EFFORT_THRESHOLD
      ),
      moneyPits: this.features.filter(f =>
        f.riceScore < VALUE_THRESHOLD && f.effort >= EFFORT_THRESHOLD
      )
    };
  }

  generateSprintRecommendation(availableCapacity) {
    const recommendation = {
      committed: [],
      stretch: [],
      deferred: [],
      totalEffort: 0
    };

    // Priority 1: All quick wins that fit
    for (const feature of this.matrix.quickWins) {
      if (recommendation.totalEffort + feature.effort <= availableCapacity) {
        recommendation.committed.push({
          ...feature,
          reason: "Quick win - high value, low effort"
        });
        recommendation.totalEffort += feature.effort;
      }
    }

    // Priority 2: Strategic projects (one per sprint typically)
    const strategicSorted = this.matrix.strategicProjects
      .sort((a, b) => b.riceScore - a.riceScore);

    for (const feature of strategicSorted) {
      if (recommendation.totalEffort + feature.effort <= availableCapacity) {
        recommendation.committed.push({
          ...feature,
          reason: "Strategic project - high value justifies effort"
        });
        recommendation.totalEffort += feature.effort;
        break;  // Usually only one strategic project per sprint
      }
    }

    // Priority 3: Fill-ins if capacity remains
    for (const feature of this.matrix.fillIns) {
      const remainingCapacity = availableCapacity - recommendation.totalEffort;
      if (feature.effort <= remainingCapacity) {
        recommendation.stretch.push({
          ...feature,
          reason: "Fill-in - only if quick wins and strategic work complete"
        });
      }
    }

    // Defer money pits and anything that doesn't fit
    const committed = recommendation.committed.map(f => f.id);
    const stretch = recommendation.stretch.map(f => f.id);

    recommendation.deferred = this.features
      .filter(f => !committed.includes(f.id) && !stretch.includes(f.id))
      .map(f => ({
        ...f,
        reason: this.matrix.moneyPits.includes(f)
          ? "Money pit - low value, high effort. Recommend NOT doing this."
          : "Good feature but doesn't fit in available capacity this sprint."
      }));

    return recommendation;
  }

  generateMarkdownReport(availableCapacity) {
    const rec = this.generateSprintRecommendation(availableCapacity);

    let report = `# Sprint Prioritization Report\n\n`;
    report += `**Available Capacity:** ${availableCapacity} person-days\n`;
    report += `**Committed Work:** ${rec.totalEffort} person-days\n`;
    report += `**Buffer Remaining:** ${(availableCapacity - rec.totalEffort).toFixed(1)} person-days\n\n`;

    report += `## ‚úÖ Committed Features (${rec.committed.length})\n\n`;
    for (const feature of rec.committed) {
      report += `### ${feature.name}\n`;
      report += `- **RICE Score:** ${feature.riceScore}\n`;
      report += `- **Effort:** ${feature.effort} days\n`;
      report += `- **Rationale:** ${feature.reason}\n\n`;
    }

    if (rec.stretch.length > 0) {
      report += `## üéØ Stretch Goals (${rec.stretch.length})\n\n`;
      report += `*Only pursue if committed work completes ahead of schedule*\n\n`;
      for (const feature of rec.stretch) {
        report += `- ${feature.name} (${feature.effort} days, RICE: ${feature.riceScore})\n`;
      }
      report += `\n`;
    }

    report += `## üìã Deferred Features (${rec.deferred.length})\n\n`;
    for (const feature of rec.deferred) {
      report += `### ${feature.name}\n`;
      report += `- **RICE Score:** ${feature.riceScore}\n`;
      report += `- **Effort:** ${feature.effort} days\n`;
      report += `- **Why Deferred:** ${feature.reason}\n\n`;
    }

    return report;
  }
}
```

## Domain-Specific Wisdom

### Common Challenges

1. **Stakeholder Pressure for Overcommitment**
   - Description: Stakeholders want more features than team capacity allows, creating pressure to over-commit
   - Approach: Show the math. Use capacity calculator to demonstrate what fits sustainably. Offer scenarios: "We can do A+B well, or we can do A+B+C poorly and risk shipping nothing."
   - Teaching: Over-commitment leads to under-delivery. Better to ship 3 features well than to fail at 5.

2. **Mid-Sprint Scope Creep**
   - Description: New "urgent" requests arrive during sprint execution
   - Approach: Protect the sprint. If something must be added, facilitate explicit conversation about what gets removed. Use equal-swap rule: new item in = committed item out.
   - Teaching: Constantly changing direction prevents completion of anything. Emergencies are rare; most "urgent" items can wait 6 days.

3. **Estimation Uncertainty**
   - Description: Team lacks confidence in effort estimates, leading to either over or under-commitment
   - Approach: Use confidence intervals instead of point estimates. Track actual vs estimated over time to calibrate. Invest in spikes for high-uncertainty items.
   - Teaching: Estimation is a skill that improves with practice and feedback. The goal isn't perfect accuracy but consistent calibration.

4. **Technical Debt Neglect**
   - Description: Pressure to ship features causes technical debt to accumulate until it cripples velocity
   - Approach: Make technical debt visible using "interest rate" metaphor (how much is this slowing us down?). Reserve 20-30% of sprint capacity for debt reduction.
   - Teaching: Technical debt is like financial debt - some is strategic, but too much is bankruptcy. Pay down the highest-interest debt first.

### Pro Tips

- üí° **Use "pizza teams"** - Keep sprint teams small (4-6 people). Larger teams spend more time coordinating than building.
- üí° **Front-load risk** - Tackle uncertain/risky work early in sprint when there's still time to adapt.
- üí° **Celebrate done, not busy** - Reward shipping complete features, not starting many things.
- üí° **Make buffer visible** - Actually put "buffer time" in the sprint plan so it doesn't get filled with "just one more thing."
- üí° **Review velocity trends, not point-in-time** - A team at consistent 12 days/sprint velocity is healthier than one oscillating between 8 and 18.
- üí° **Write deferral reasons down** - When you defer a feature, document why. This prevents it from coming back as "urgent" next sprint without new information.
